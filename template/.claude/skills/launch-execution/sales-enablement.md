## 1. Demo Script

### Structure: The Problem-Solution-Proof Framework

```markdown
# Demo Script: [Feature Name]

## Demo Setup (2 minutes)
**Environment:** [Link to demo environment]
**Persona to use:** [Role, e.g., "Program Manager at a 500-person org"]
**Pre-configured data:** [What's already set up]

## Opening: Set the Stage (1 minute)
"I want to show you something that [persona] at [similar company] has been struggling with..."

**Hook statement:**
"[X%] of teams we talk to struggle with [problem]. The reason is usually [root cause]. Let me show you how [feature] solves this."

---

## The Demo Flow (5-7 minutes)

### Scene 1: Show the Problem (2 minutes)
1. Navigate to [starting screen]
2. Show [current state / pain point]
3. Narrate: "Right now, [persona] has to [manual process]. This means [negative outcome]."

### Scene 2: Introduce the Solution (3 minutes)
1. Navigate to [new feature]
2. Demonstrate [key capability 1]
   - "Watch what happens when I [action]..."
   - "This gives you [benefit]"
3. Demonstrate [key capability 2]
   - "Now notice how [feature] automatically [outcome]..."
   - "This saves [time/cost/risk]"

### Scene 3: Show the Result (2 minutes)
1. Navigate to [outcome screen]
2. Show [before/after or result]
3. Narrate: "In [time period], [persona] went from [before state] to [after state]. That's [quantified benefit]."

---

## Closing: The Ask (1 minute)
"Based on what you've seen, what questions do you have about how this would work for [prospect's situation]?"

**Transition to discovery:**
"To make sure this is a good fit, can you tell me about [discovery question]?"

---

## Demo Do's and Don'ts

| Do | Don't |
|----|-------|
| Start with the problem, not the feature | Open with "Let me show you this cool feature" |
| Use customer's language and examples | Use internal terminology |
| Focus on one key benefit | Show everything the feature can do |
| Pause for questions and reactions | Rush through to hit every screen |
| Connect to their specific situation | Give a generic, memorized pitch |
| End with a clear next step | Trail off without direction |
```

## 2. Talk Tracks by Persona

### Executive Talk Track (C-Suite, VP)

**Focus:** Business outcomes, strategic value, risk reduction

```markdown
# Executive Talk Track: [Feature]

## Opening (30 seconds)
"Executives at [similar companies] tell us their biggest concern with [domain] is [strategic problem]. [Feature] directly addresses this by [unique value]."

## Key Messages
1. **Strategic Value:** "[Feature] enables [strategic outcome], which matters because [business context]."
2. **Risk Reduction:** "Companies using [feature] have seen [risk metric improve by X%]."
3. **Competitive Position:** "This capability is [differentiator vs. competitors]."

## Close
"What would it mean for your organization if you could [outcome]?"

## If They Ask About ROI
"Based on [data/source], customers typically see [ROI metric]. But the real value is [strategic benefit]."
```

### Manager/Practitioner Talk Track

**Focus:** Day-to-day impact, workflow improvement, time savings

```markdown
# Practitioner Talk Track: [Feature]

## Opening (30 seconds)
"Teams like yours spend [X hours] per week on [painful task]. [Feature] automates this so you can focus on [higher-value work]."

## Key Messages
1. **Time Savings:** "Instead of [manual process], you'll [automated process]. That's [X hours] saved per week."
2. **Workflow Improvement:** "The feature integrates with [existing workflow], so there's no disruption."
3. **Immediate Value:** "You can start seeing results in [timeframe]."

## Close
"Walk me through your current process for [task]. Where does it break down?"
```

### Technical/Evaluator Talk Track

**Focus:** How it works, integration, security, scalability

```markdown
# Technical Talk Track: [Feature]

## Opening (30 seconds)
"From a technical perspective, [feature] [key technical benefit]. Let me explain how it works."

## Key Messages
1. **Architecture:** "[Feature] is built on [technical foundation], which means [reliability/performance benefit]."
2. **Integration:** "It integrates via [method] with [systems]. Setup takes [time]."
3. **Security:** "Data is [security posture]. We [security approach]."
4. **Scalability:** "We've tested with [scale metric] and performance remains [result]."

## Close
"What technical requirements do I need to know about for your environment?"
```

## 3. Objection Handling Guide

### The LAER Framework

**L**isten → **A**cknowledge → **E**xplore → **R**espond

```markdown
# Objection Handling: [Feature]

## Objection 1: "We already use [alternative]."

### Listen
[Let them explain what they use and why]

### Acknowledge
"[Alternative] is a solid choice for [what it does well]. A lot of our customers used to use it."

### Explore
"What's working well for you? Where are you seeing gaps?"
[Probe for pain points the alternative doesn't solve]

### Respond
"That makes sense. Where we see customers like you get value is [unique capability]. Unlike [alternative] which [their limitation], we [our advantage].

For example, [customer name] was using [alternative] but switched because [specific pain point]. With [feature], they were able to [outcome]."

### Proof Point
"[Customer quote or metric]"

---

## Objection 2: "It's too expensive / We don't have budget."

### Acknowledge
"Budget is always a consideration. Let me make sure we're comparing apples to apples."

### Explore
"What's driving the budget conversation? Is it the timing, or the value proposition?"

### Respond
**If timing:** "I understand. Let's talk about [pilot/trial/start small approach] so you can validate the value first."

**If value:** "Let me break down the ROI. [Feature] typically saves [X hours] per week, which at [hourly rate] is [$Y] annually. Plus [additional benefit]. So the payback period is [timeframe]."

---

## Objection 3: "We need to think about it / check with the team."

### Acknowledge
"Of course. This is an important decision."

### Explore
"What specific questions does the team need to answer? What would make this a clear yes?"

### Respond
"Let me help you with that. I can [provide specific resource / schedule technical call / share case study] to address [specific concern]."

---

## Objection 4: "This doesn't do [missing feature]."

### Acknowledge
"You're right, that's not in [feature] today."

### Explore
"How critical is [missing feature] to your use case? How often would you need it?"

### Respond
**If roadmap:** "[Missing feature] is on our roadmap for [timeframe]. Here's how customers are handling it in the meantime: [workaround]."

**If not planned:** "Fair point. For customers who need that, we typically see them [alternative approach]. But for your primary use case of [use case], [feature] is specifically designed to [unique value]."

---

## Objection 5: "We're happy with our current vendor."

### Acknowledge
"It sounds like [current vendor] is working well for you."

### Explore
"What's kept you with them? What would make you consider a change?"

### Respond
"That's great that it's working. Most of our customers weren't looking to switch either—they just found that as their needs grew, [current vendor] couldn't [limitation].

I'm not suggesting you switch everything. But for [specific use case], [feature] might give you [unique benefit]. Worth a conversation?"
```

## 4. ROI Calculator Template

```markdown
# ROI Calculator: [Feature]

## Customer Inputs
| Input | Value | Notes |
|-------|-------|-------|
| Team size | [#] | Number of users |
| Hours/week on [painful task] | [#] | Current manual effort |
| Fully loaded hourly cost | $[#] | Salary + benefits + overhead |
| [Domain] incidents per month | [#] | Errors, delays, escalations |
| Cost per incident | $[#] | Rework, customer impact |

---

## Calculated Savings

### Time Savings
- Hours saved/week: [Team size] × [Hours] × [Efficiency gain %]
- Weekly savings: [Hours saved] × [Hourly cost]
- Annual savings: [Weekly savings] × 52

### Error Reduction
- Incidents reduced/month: [Current incidents] × [Reduction %]
- Monthly savings: [Incidents reduced] × [Cost per incident]
- Annual savings: [Monthly savings] × 12

### Total Annual Value
[Time savings] + [Error reduction] = **$[Total]**

---

## Payback Period
[Feature cost] / [Monthly value] = [X] months

---

## Proof Points
- [Customer X] achieved [specific savings]
- [Industry benchmark] shows [typical improvement]
```

## 5. Discovery Questions

```markdown
# Discovery Questions: [Feature]

## Current State
- "Walk me through how you currently handle [process]."
- "What tools are you using today for [domain]?"
- "How much time does your team spend on [painful task]?"
- "Who owns [process] today? Who else is involved?"

## Pain Points
- "Where does [process] break down today?"
- "What's the most frustrating part of [current approach]?"
- "What happens when [problem occurs]?"
- "How do you measure success for [domain] today?"

## Impact
- "How much is [problem] costing you in [time/money/risk]?"
- "What would change if you could [desired outcome]?"
- "How does [problem] affect other teams or processes?"
- "What's the risk of not solving this?"

## Buying Process
- "Who else needs to be involved in this decision?"
- "What's your timeline for making a change?"
- "What would need to be true for you to move forward?"
- "Have you evaluated other solutions? What happened?"

## Qualification Signals
✅ **Good fit:**
- Explicit pain with current approach
- Quantified impact (time, money, risk)
- Decision timeline within 90 days
- Budget allocated or accessible
- Champion identified

❌ **Poor fit:**
- No clear pain point
- "Just looking" / no timeline
- Multiple competitors in parallel evaluation
- No budget discussion in 2+ calls
- No internal champion
```
